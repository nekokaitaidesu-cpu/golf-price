# golf-price 運用ガイド（Claude向け）

メルカリ/楽天のゴルフクラブ相場を集計し、割安出品の発見（今日の本命）と
LINE通知（本命スナイパー）を回すプロジェクト。モデル（Fable/Opus等）に
依存しない手順書。過去の判断例・教訓は `honmei_notes/` の日次メモと
auto-memory（honmei-genbutsu-check / flip-inventory）を必ず参照。

## 「今日の本命教えて」の手順（所要 約20分）

1. **同期**: `git fetch origin data --depth 1` →
   `git checkout origin/data -- history.db .cache` → `git reset --quiet`
   （クラウドが1日8回更新。popularity.json の generated_at を確認）
2. **スキャン**: `python scripts/honmei_scan.py`（約10分・バックグラウンド推奨）
   → ライブ再集計を popularity.json にマージ＋一次候補を
   `.cache/honmei_candidates.json` に出力
3. **一次トリアージ**（候補リストの時点で落とすもの）:
   - タイトルに シャフト/カバー/ヘッド 系の語 → 部品単品疑い
   - 「名器！」出品者 → ヘッド単品専門業者（実績4件）
   - 短尺・訳あり・改造表記、レフティ（市場が別）
   - アイアンのセット本数が中央値の基準（6本フル）と不一致
   - 既知の売れ残り個体（前日メモと突合）
   - 🔨オークション印 → 即決不可。通知・提案時に必ず明記
4. **検死**: `python scripts/honmei_autopsy.py <id>...` で説明文・いいね・写真取得
   → **写真は必ずReadで目視**。チェック順: ①ホーゼルが空いていないか
   （説明文無言のヘッド単品が実在） ②フェース・ソール・クラウンの傷
   ③セット物は本数を数える。「いいね多数×売れ残り」は割安でない証拠として減点
5. **楽天照合**: history.db の当日 used_min × ライブ実売中央値で
   粗利 = 中央値×0.9 − 送料1,500 − used_min。**回転（7日売れ数）のない粗利は罠**
6. **メモ保存**: `honmei_notes/YYYY-MM-DD.md`。フォーマットは前日のメモを踏襲
   （データ基準の時刻注記・本命・検死記録表・楽天本命・傾向・教訓・所感）

## 判断基準（2026-07時点の運用値）

- 割安 = 実売中央値の35〜72%、準割安 = 〜80%。35%未満はほぼ部品
- ウォッチ機種 = 30日15本以上 or 7日4本以上売れ、中央値8,000円以上
- 転売判定と自分用判定は分けて書く。オークションは
  「実売中央×0.9 − 送料」から上限を先に決めて機械的に
- 中央値が急変した機種はヘッド単品混入をまず疑う
- アイアンは回転（7日売れ数）を最優先。夏は原則手出し無用

## 部品単品との戦い（イタチごっこの記録）

検出は `golf_price/normalize.py`（タイトル: detect_head_only / is_parts_junk、
説明文: detect_head_only_desc）。すり抜けを見つけたら:
1. パターン/トークンを追加（誤検出側のテストも必ず書く — 完品を落とす方が痛い）
2. `python -c` でテストしてから commit & push（クラウド集計も同じコードを使う）
3. その日のメモの「教訓」と auto-memory に記録

テキスト検出の限界: 説明文に何も書かないヘッド単品が実在するため、
**購入判定の最終関門は常に写真のホーゼル目視**。

## LINE通知（本命スナイパー）

`notify_honmei.py` がタスクスケジューラで定期実行。通知全文は `notify.log` に
残る（「LINEで来たあれ何？」はここを引く。旧ログは文字化けあり）。
通知には 🔨オークション・⚠シャフト表記 の警告が付く。
ボットは「同モデルのより安い売れ残り」までは見ない — 通知物件の購入相談を
受けたら、必ず同モデルの販売中と突合してから判定すること。
